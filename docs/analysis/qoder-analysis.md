# Qoder Wiki 能力分析

> 分析对象: URP_Tamplate 项目的 Qoder 生成的 Wiki
> 生成文件: 88 个 (87 页面 + 1 meta JSON)

---

## 一、Qoder 做了什么

### 1. 源文件分析

扫描整个项目的源码和配置文件，不限于 .cs，还包括 .md、.json、.asset、Python 等。

| 文件类型 | 数量 |
|:---------|:----:|
| Wiki 页面 | 87 个 |
| 目录分类 | 256 个（catalogs） |
| 知识关系 | 68 条（relations） |

### 2. 分类体系（256 个 catalogs）

每个 catalog 有：id、name（中文名）、description（英文标识）、prompt（LLM 生成指引）。覆盖 15 个主题域：

```
项目概述 / 核心架构设计 / UI系统架构 / 交互系统设计 / 全局管理器系统
控制器系统 / 数据管理系统 / 多语言和本地化 / 渲染系统详解 / 资源管理
API参考文档 / 性能优化和最佳实践 / 故障排除和调试 / 部署和发布 / 配置和设置
Python后端集成 / 扩展开发指南 / 项目工具和脚本
```

### 4. 知识图谱（68 条 relations）

metadata.json 中的 `knowledge_relations` 记录了页面间的 PARENT_CHILD 关系。

### 5. 源文件行级溯源

每页的每个章节都标注了源文件的起止行号，例如：

```
**章节来源**
- [Singleton.cs:11-135](file://Assets/Scripts/Singleton/Singleton.cs#L11-L135)
- [GlobalDataMgr.cs:14-141](file://Assets/Scripts/GlobalManagement/GlobalDataMgr.cs#L14-L141)
```

## 二、Qoder 与 Baize Wiki 的对比

| 能力 | Qoder | 我们 |
|:-----|:-----|:-----|
| 代码扫描 | ✅ 全文本扫描 | ✅ `--scan-all` |
| 目录结构 | ✅ Level 2~3 | ✅ Level 1/2/3 |
| **LLM 摘要** | ✅ **每页 50-100 字引言** | ⚠️ 首段提取（Local）/ 待做（Remote） |
| **Mermaid 架构图** | ✅ **每个页面 2-5 张图** | ❌ 未实现 |
| **类图/流程图/时序图** | ✅ **多种图表类型** | ❌ 未实现 |
| **依赖分析** | ✅ **跨文件关系描述** | ❌ graph.json 有节点无关系 |
| **性能建议** | ✅ **每页都有** | ❌ 未实现 |
| **故障排查** | ✅ **每页都有** | ❌ 未实现 |
| **行级溯源** | ✅ **每节 file:line** | ❌ 未实现 |
| **源文件引用** | ✅ `<cite>` 区块 | ❌ 未实现 |
| **知识图谱** | ✅ **68 条 PARENT_CHILD 关系** | ⚠️ graph.json 有节点无关系 |
| **分节锚点导航** | ✅ **每页目录** | ❌ 未实现 |
| 中文支持 | ✅ 全中文 | ⚠️ 部分中文 |
| 生成后端 | 远程 LLM（通义千问） | Local / 待做 Remote |

## 三、差距摘要

Qoder 做了 **12 件事**，我们只做了 **3 件**（扫描、目录、摘要初版）。

核心差距在于：**Qoder 每页都调了 LLM 做深度理解**，而我们目前的 Local 编目只是首段提取。Qoder 的每页内容相当于我们文档里规划的 **🥇 Level 3 图谱层** 加上大量 LLM 写的分析文本，不是简单的元数据。
