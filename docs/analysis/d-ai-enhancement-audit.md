## 🔴 方向性问题

### 1. 输入结构缺少 ContentHash，增量更新缺少锚点

**原始设计（第 118-125 行）：**
```go
FileInfo {
    path: "核心架构/核心架构.md"
    extension: ".md"
    content: "..."
    frontmatter: { title: "...", tags: [...] }
}
```

**问题：** 增量更新场景下，系统怎么知道这个文件"变了"？目前只定义了处理流程（"只处理变更文件"），但没有定义变更检测的机制。如果每次 build 都重新跑所有文件的 LLM，增量就失去了意义。

**建议：** 输入结构补充两个字段：

```go
FileInfo {
    // ... 已有字段
    size: int64           // 文件大小，用于截断判断
    content_hash: string  // 内容哈希（SHA256），增量检测锚点
}
```

同时 `Page` 模型增加 `llm_hash` 字段，记录生成摘要时的 ContentHash 快照。增量构建时，对比文件当前 hash 与 Page 记录的 hash，一致则跳过 LLM 调用。

---

### 2. Level 3 缺少增量更新策略

**原始设计（第 300-380 行）：** 详细定义了分批+合并轮的架构，但所有描述都基于**首次全量构建**。Level 3 的增量更新比 Level 2 复杂——新增一个文件可能影响已有的实体关系，需要重新做部分关系发现。

**具体场景未覆盖：**

| 场景 | 影响 | 风险 |
|------|------|------|
| 新增文件 | 新增实体可能与已有实体有关系 | 现有关系图中缺了新增的边 |
| 修改文件 | 实体可能变更（新增/删除/改名） | 涉及变更实体的关系全部需要重审 |
| 删除文件 | 实体会消失 | 关系图中的 dangling 边 |

**建议：** Level 3 增量策略分三种情况定义：

```
新增文件 → 跑 Level 2（含实体）→ 全局实体集合并 → 只对新实体的关系做第二轮分析
修改文件 → 重跑 Level 2 → 全局实体集更新 → 只对涉及变更实体的关系重做第二轮
删除文件 → 从全局实体集移除 → 移除涉及的关系 → 更新图谱
```

> 核心原则：**谁变了就只重算谁**，不因一个文件变化而重跑整个图谱。

## 🟡 设计缺口

### 3. 错误处理缺少敏感信息过滤

**原始设计（第 218-230 行）：** 定义了 5 种错误处理（格式错误、空文件、超长、超时、网络中断），但缺少一个关键的**安全步骤**。

**问题：** LLM 返回的摘要如果包含了代码中的密钥（AK/SK）、内网 IP、数据库连接串，Agent 直接消费这个摘要，敏感信息就以更显眼的方式暴露了。摘要比原文更危险——原文在角落里，摘要在开头。

**建议：** 在后处理阶段增加敏感信息过滤步骤（第 ④ 步）：

```
④ 后处理
    ├── 验证 JSON 完整性
    ├── 敏感信息过滤 ← 新增
    │   ├── 密钥类: (?:AK|SK|secret|token|password|key)[:=]["'][A-Za-z0-9_\-]{16,}["']
    │   ├── 内网 IP: 10\.\d+\.\d+\.\d+ | 172\.(1[6-9]|...)
    │   └── 匹配项替换为 [REDACTED]
    ├── 截断摘要 ≤ 100 token
    ├── 去重关键词
    └── 写入 Page
```

---

### 4. 截断策略的假设不成立

**原始设计（第 225 行）：** "文件 > 1MB（代码）— 截断后 500KB，LLM 看尾部（关键信息通常在最后）"

**问题：** 对于代码文件，关键信息（类定义、接口声明、导出符号）几乎都在**开头**或头部注释。Go 的 `package` 声明、C# 的 `namespace`、Python 的 imports——都在文件前 100 行。看尾部会漏掉最重要的结构信息。

**建议：** 按文件类型区分截断策略：

```
代码文件（.cs/.go/.py/.js）→ 截取前 500KB（类定义在头部）
文档文件（.md/.mdx）      → 截取前 400KB + 尾 100KB（结论可能在尾部）
配置文件（.json/.yaml）   → 截取前 500KB（不熟悉这类文件的尾部结构）
```

---

### 5. 并发控制缺少全局协调

**原始设计（第 195 行）：** "Remote: 逐个调用，支持并发 (concurrency=5)"

**问题：**
- 并发数 5 硬编码，不同 API 的限制不同（OpenAI 3500 RPM vs Anthropic 1000 RPM）
- 没有 Rate Limiting——如果 5 个并发同时被限流（429），退避是否需要协调？
- 5 个并发对于某些 API 过于保守（浪费），对某些 API 又可能触发限流

**建议：** 改为全局 Rate Limiter：

```
所有并发共享同一个 Rate Limiter
  - OpenAI:    根据 tpm/rpm 自动计算并发上限
  - Anthropic: 根据 requests_per_minute 计算
  - Local:     串行（默认），可通过 --llm-concurrency 放开
  
Rate Limiter 行为：
  - 收到 429 且含 Retry-After → 所有并发等待指定时间
  - 连续 429 → 逐步降低并发数
  - 恢复正常 → 逐步恢复
```

### 6. Level 2 产出中缺少 entities 字段设计

**原始设计（第 140-151 行）：** Level 2 输出 `{ summary, keywords }`，Level 3 的第一轮才提取 entities。但文档又说"与 Level 2 合成一轮调用"——这意味着 Level 2 实际上已经做了实体提取，只是输出里没有这个字段。

**问题：** 如果 Level 2 输出不包含 entities，用户先跑 Level 2，之后再升级到 Level 3 时，需要**重新调用 LLM** 才能拿到实体数据——浪费了之前的 LLM 成本。

**建议：** Level 2 的输出定义直接包含 entities：

```go
type Page struct {
    // ... 已有字段
    Abstract  string     // 50-100 token 摘要
    Keywords  []string   // 3-10 个关键词
    Entities  []Entity   // ← 新增，Level 2 一并产出，Level 3 直接复用
}

type Entity struct {
    Name string  // "Singleton<T>"
    Type string  // class | interface | function | module | concept | data
    Role string  // defined | imports | implements | uses
}
```

这是"埋种子"的设计——Level 2 多花一点 token 产出 entities，Level 3 升级时零额外成本。

---

### 7. Level 3 的查询接口需要定义

**原始设计（第 381-420 行）：** 定义了产出物（每页依赖关系 + 根目录图谱概览 + repowiki-graph.json），但 Agent 怎么查询图谱？

**问题：** 图谱数据写入了 JSON 文件和 Markdown，但 Agent 无法通过 MCP 直接查询"Singleton<T> 被谁依赖"或"从 CameraSystem 到 UISystem 的最短路径"。Agent 需要自己解析 JSON 文件——体验割裂。

**建议：** 新增 MCP 工具 `wiki_graph`：

| 操作 | 参数 | 返回 |
|------|------|------|
| `entities` | (无) | 所有实体列表及所在文件 |
| `relations` | entity, depth | 指定实体的关系网 |
| `path` | entity, target_entity | 两实体间的最短依赖路径 |
| `layers` | (无) | 架构分层结构 |

这 4 个操作覆盖了 Agent 探索知识图谱的典型需求。

## 💡 优化建议

### 8. Prompt 语言参数化

**原始设计（第 161-215 行）：** Prompt 模板中写死了"中文摘要 / 中文关键词"。

**建议：** 将摘要语言改为 `{{lang}}` 参数，默认与源文件语言一致，用户可通过 `--summary-lang` 覆写。原因：
- 源文件可能是英文，LLM 产出中文摘要，Agent 读起来不协调
- 多语言项目的用户可能希望摘要语言统一

---

### 9. 代码文件 Prompt 增加关注点引导

**原始设计（第 164-178 行）：** 代码文件 prompt 内容为"概括这个文件的职责和核心逻辑"。

**建议：** 增加优先级引导，帮助 LLM 聚焦重点：

```
请按以下优先级理解：
1. 这个文件定义了哪个核心类型（类/接口/模块/函数）？
2. 这个类型的主要职责是什么？
3. 它公开了哪些关键方法/API？
4. 它依赖了哪些外部模块或类型？
```

如果文件是工具/辅助类（没有核心类型），则提示 LLM 重点概括功能范围而非实现细节。

---

### 10. 预留自动 Level 路由的接口

**原始设计（第 445 行+）：** 目前通过 `--catalog-level` 手动选择层级。

**建议：** 在设计层面预留自动路由的可能，不一定现在实现，但接口要考虑：

```
Agent 查询 → 系统自动判断用哪层回答
  简单事实查询 → Level 1（目录定位 → 读原文的两三行）
  概念理解查询 → Level 2（摘要 → 判断是否深入）
  关系探索查询 → Level 3（图谱 → 关系链 → 定向阅读）
```

这个在规划阶段只需要在文档里提一句"预留此方向"，不需要具体设计。
