## 2. 残留的微调问题

以下问题不需要再改文档即可进入编码，仅在编码时留意即可。

### 2.1 cli-spec.md JSON 输出示例含 link 字段（残留）

**文件：** `cli-spec.md` 第 275-293 行

**问题：** JSON 输出示例仍包含 `"links": 47, "dangling_links": 3, "dangling_links_file"`。

**影响：** 极低。功能上这只是一个示例，实际输出的 JSON schema 由 Generator 定义。但示例若与实际不符，可能误导。

**建议：** 将 JSON 示例中的 `links`/`dangling_links` 替换为仅 Phase 1 包含的字段：
```json
"summary": {
  "total_files": 24,
  "parsed": 24,
  "pages": 24,
  "directories": 8
}
```

### 2.2 mcp-spec.md wiki_build 输出示例含 `"links": 47`（残留）

**文件：** `mcp-spec.md` 第 82 行

**问题：** MCP wiki_build 的输出示例中 `"summary":{"pages":24,"directories":8,"links":47}` 包含 `links`。

**影响：** 极低。同 2.1，仅为示例。

**建议：** 移除 `"links":47`。

### 2.3 architecture.md 第 5 节有多余的残留文本

**文件：** `architecture.md` 第 296-308 行

**问题：** "二进制探测算法"标题后有一个孤立的 `▼` 符号和一段不连续文本：

```
### 5.3 二进制探测算法
    ▼
识别文件类型并解析
    │
    ├── .md       → Markdown 解析 ...
```

这看起来是在文档迭代中未清理干净的格式化碎片。

**建议：** 删除第 296-308 行的残留内容，使二进制探测算法的代码块直接跟随第 5.2 节结束处。

### 2.4 data-model.md Frontmatter 代码块有语法错误

**文件：** `data-model.md` 第 95 行

**问题：** 兼容性说明插入在代码块中间，导致 Go struct 定义在语法上被切断：

```go
    Custom      map[string]any    `yaml:",inline"` // ...
}
> ⚠️ 兼容性说明...
}
```

第 95 行的兼容说明使用了 `>` 引用语法插在 `}` 之后，但 Go struct 在 `}` 处已结束。第 96 行多了一个 `}`。

**建议：** 将兼容性说明移到代码块外部，删除第 96 行多余的 `}`。调整后结构应当是：代码块完整结束 → 引用块说明。

### 2.5 architecture.md 的 Section 5 仍引用 Phase 1 不处理的内容

**文件：** `architecture.md` 第 296-308 行中的文件类型匹配列表包含 `.json`、`.yaml`、`.go`、`.py` 等。

**问题：** Phase 1 默认模式只扫描 `.md/.mdx`，所以这些文件类型在当前 Phase 不会进入 Parser。但流程图容易让人误以为 Phase 1 会解析这些文件。

**建议：** 在文件类型列表上方增加注释 "Phase 2+ 启用 --scan-all 后生效"。
