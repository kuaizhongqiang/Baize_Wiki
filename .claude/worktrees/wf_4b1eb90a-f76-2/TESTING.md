# Tester 工作规范

> 适用于负责 Baize Wiki 测试的 Testing Agent。

## 工作流程

1. **Issue 驱动**
   - 测试任务由 PM 通过 Issue 分配，开工前确认 Issue 已 Assign
   - 测试方案遵循 PM 指定的范围，不自行扩大或缩小

2. **测试发现 vs Bug 确认**
   - 执行测试前先用 `go test ./...` 确认基线状态，记录哪些已失败
   - 测试发现的失败分两类处理：
     - **测试代码错误**（期望值写错）→ 修正测试，不报 bug
     - **生产代码错误**（行为不符合规范）→ 报 bug Issue，附复现步骤和根因分析
   - 不确定归属时，先报 bug 由 PM 裁定

3. **Bug 报告规范**
   - 标题：`🐛 <package>: <具体问题>`
   - 内容包含：
     - 复现步骤（输入 → 期望 → 实际）
     - 根因分析（哪个文件、哪行逻辑）
     - 影响范围（哪些功能受影响）
     - 优先级建议（P0/P1/P2）
   - 一个 Issue 只报告一个问题，不合并

4. **回归验证**
   - 开发修复后，在对应 Issue 评论区验证结果
   - 验证通过标记 `✅ verified`，不通过附复现证据
   - 验证时需在修复分支上重新运行全部测试

5. **测试代码规范**
   - 不修改生产代码，仅添加 `_test.go` 文件
   - 遵循项目现有风格：表格驱动 + `testify/assert` + `testify/require`
   - 新增测试文件按职责命名：`<package>_boundary_test.go`、`<package>_security_test.go` 等

6. **覆盖率要求**
   - 每次测试任务完成后报告新增测试数和 `go test -cover` 覆盖率变化
   - P0 包（storage/、app/）覆盖率达到 80%+
   - 核心逻辑包（parser/、generator/）覆盖率达到 70%+

7. **向 PM 报告**
   - 每轮测试完成后提交执行报告，包含：
     - 新增测试数 / 总测试数
     - 通过/失败统计
     - 覆盖率变化
     - Bug 清单（编号 + 状态）
